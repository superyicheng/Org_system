# Shared cloud deployment

This is the production path for a team: one HTTPS Org_system service, one shared PostgreSQL database, Google identity in the browser, and one revocable Codex token per employee laptop. No employee runs the database or exposes their laptop.

## 1. Prepare Google sign-in

In Google Cloud Console, create an OAuth 2.0 **Web application** client for Google Identity Services. Add the exact deployed HTTPS origin (for example `https://org-system.onrender.com`) to its Authorized JavaScript origins. Copy the resulting `GOOGLE_CLIENT_ID`; it is safe to expose to the browser, but never commit operational secrets.

Choose the approved employee domain and the boss/admin email before deployment:

```text
GOOGLE_WORKSPACE_DOMAIN=example.com
ORG_SYSTEM_ADMIN_EMAILS=boss@example.com
```

Leave `GOOGLE_WORKSPACE_DOMAIN` empty only if intentionally allowing any verified Google account.

## 2. Deploy the repository

The repository includes a production Docker image, [compose.yaml](../compose.yaml), and [render.yaml](../render.yaml). The fastest hosted route is to create a Render Blueprint from this GitHub repository; it provisions the web service and PostgreSQL database. Set these service variables in the host dashboard before the first successful start:

```text
GOOGLE_CLIENT_ID=<Google web client id>
GOOGLE_WORKSPACE_DOMAIN=example.com
ORG_SYSTEM_ADMIN_EMAILS=boss@example.com
```

`SESSION_SECRET` is generated by the included Render Blueprint. The Blueprint derives its default `PUBLIC_URL` and `ALLOWED_ORIGINS` from Render's external hostname. If you add a custom domain, override both with that exact `https://` origin. On another host, generate a long random session secret and set all three values yourself. The service accepts the ordinary `postgresql://…` connection string provided by most hosts and converts it to the installed PostgreSQL driver automatically.

For a self-hosted or local shared instance, copy `.env.example` to `.env`, replace every placeholder, then run:

```bash
docker compose up --build
```

The app is at `http://127.0.0.1:8000` in that setup. Use a real HTTPS hostname for employee access and set `PUBLIC_URL` and `ALLOWED_ORIGINS` to that hostname.

## 3. Verify the shared service

After deployment:

1. Visit `https://your-service/health`. It must return `"status":"ok"` and `"auth_mode":"google"`.
2. Visit the root page in a browser. Sign in with an allowed Google account.
3. Confirm an employee cannot open **Admin health** and only sees their own private experiences; the configured admin can access Admin health.
4. Use **Create Codex connection** in the signed-in page. Each person gets a separate token that is stored only as a hash in PostgreSQL.
5. Follow [Codex employee setup](CODEX_EMPLOYEE_SETUP.md) on one employee laptop and ask Codex to call `recall_experience`.

An unauthenticated request to `POST /mcp/` must return `401`; that is expected. The employee's bearer token is the credential that makes the shared MCP service usable from their own Codex client.

## Operations notes

- The app seeds transparent sample records only into a brand-new empty database. Start with a fresh production database if you do not want hackathon fixtures available.
- Browser sessions expire after eight hours. Employees can sign in again; their Codex token remains valid until revoked.
- Revoke a compromised laptop token from the same signed-in page immediately. Revocation takes effect on the next MCP request.
- Back up the managed PostgreSQL database according to the hosting provider's backup policy. Do not treat the UI as the only copy of organizational knowledge.
